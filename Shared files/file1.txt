                                    <span id="maincontent"></span><h2 class="main">Programming Assignment 3 # TCP File Distribution System </h2><div id="intro" class="box generalbox boxaligncenter"><div class="no-overflow"><p align="center"><b>CSCI 4273/5273: Network Systems<br /> <br /> Programming Assignment #3:</b></p>

<p align="center"><b>TCP File Distribution System</b></p>

<p align="center"><b>due <b><span>November 16<sup>th</sup> by 11:55pm</span></b></b></p>

<p>The purpose of this assignment is to build a real distribution file application using TCP sockets. You will implement a protocol to allow clients to connect to a server and join a "swarm" of hosts to download and upload from each other simultaneously as showing in figure 1. The server must manage all clients’ requests and contain a hash table of distributed files in all clients.</p>

<p><img src="http://moodle.cs.colorado.edu/pluginfile.php/8116/mod_assign/intro/PA3-fig.jpg" width="703" height="422" alt="PA3-sys-arch" style="display: block; margin-left: auto; margin-right: auto;" /></p>

<p align="center">Figure 1 File Distribution System</p>

<p> </p>

<p> </p>

<p><b>The clients <br /> </b>Figure 1 shows an example of how the server and clients that you will be building should communicate. The server and each client should be separate processes that are run in different console windows. The server should be listening for client connections on a TCP port.</p>

<p>Each client will be identical, running the same client code. At least 4 clients should be able to connect to the server and begin sending and receiving messages. Since the accept() function blocks you will need to use threads inside the server, as well as the 'select' function.</p>

<p>A client who wants to search/download/upload his files must connect to a server that works as a coordinator. Then the client sends to the server its name and a list of available files that he is willing to share in a directory called “Shared files”. The client’s name should be unique so it doesn’t duplicate with other clients. The server will build a hash table that contains the following fields:</p>

<p><b>&lt;file name&gt; &lt; file size&gt; &lt;file owner&gt; &lt;file owner’s IP address&gt;</b></p>

<p>The first three fields will be visible to all clients when they send a “List” request. The last one is only visible to the server.</p>

<p>After starting the connection with the server and sending a list of available files to share, the client can perform the following actions:</p>

<p>1- <b>List</b><br /> Asking the server to send a list of available files.  When the list is received from the server, the client should output:</p>

<p><b>Received list of files from server</b><br /> <b>File name || File size KB || File owner </b></p>

<p><b>            Foo1                   || 524          || Rick       </b></p>

<p><b>           Notes.txt             || 300          || Eric        </b></p>

<p><b>            Foo2                   || 2045        || Murad      </b></p>

<p><b>            Dog-picture.jpg    || 524          || Eric     </b></p>

<p> </p>

<p>2- <b>Get</b> <b>&lt;file&gt;</b></p>

<p>Request to download a file seen in the hash table.</p>

<p>3- <b>SendMyFilesList</b></p>

<p>Send to the server a list of files the client is willing to share from his directory “Shared files”</p>

<p><b> </b></p>

<p><b>The server</b></p>

<p>The server should be able to communicate with more than one client at the same time. When the server receives a new connection from a client, it will update its hash table with the client’s name, list of files, and IP address. Then the server must “Push”/send the new updated hash table to all clients it is connected to. The server DOESN’T have files. It just works as a coordinator to forward clients’ requests of files to other clients that actually have these files.</p>

<p>The server should response to the client’s commands with the following actions:</p>

<p>1- After receiving “<b>List</b>” command, the server will send the “last updated” hash table above to the client.</p>

<p>2- After receiving <b>“Get &lt;file&gt;”</b> command, the server will send to the client the IP address of the owner of this file. The client after receiving the IP address will start a new TCP connection with the file’s owner and send a request to get the desired file. All this should happen in the back scene. Your output should only show:<br /> <b>&gt; Start downloading file &lt;file name&gt; from &lt;owner&gt;</b><br />  and after downloading the whole file <br /> <b>&gt; finished downloading file</b></p>

<p> The file should be downloaded in a directory called “Downloaded files”. The client that sends the file should output, while sending:</p>

<p><b>&gt; Sending file &lt;file name&gt; to client &lt;client name&gt;</b></p>

<p>3- When server receives “SendMyFilesList” command and the data attached to it (list of files), it should update its hash table to include the files available from this client and delete any files not in the list sent by the client. The new hash table will include the client’s list of files and the other clients’ lists as well.</p>

<p>When the server accepts a new client connection, it should output<br /> <b>&gt; &lt;Client name&gt; just started connecting. This is the updated file list from all clients: <br />             File name || File size || File owner || file owner’s IP address</b></p>

<p><b>            Foo1                    || 524          || Rick       || 127.0.0.1</b></p>

<p><b>            Foo2                     || 2045        || Murad    || 165.26.15.1</b></p>

<p><b>            Dog-picture.jpg    || 524          || Eric       || 127.2.0.1</b></p>

<p><b> </b></p>

<p><b>A requirement for 5273 students:</b></p>

<p>If there is a change in the files list, The the server must “Push”/send the new updated hash table to all clients it is connected to.</p>

<p> </p>

<p><b>Sending file from the clients</b>: <br /> After the clients starts a connection with the server and sends its list of files to share. It should be available to receive any <b>Get &lt;file&gt;</b> requests from the other clients that have been forwarded from the server. So the client can work as a “client” when it connects to the server and downloads files from other users. Or it can work as a “server” to send files to other clients when they request these files. After a client sends a file to another client, both should tear down the connection.</p>

<p>If the client is currently sending a file and receive a request for a file from another machine, the machine requesting the file should receive a <i>blocking</i> message saying “The machine hosting this file is not available at this moment, please wait a little and try another time or get the file from another machine”. The machine requesting the file should drop the connection and wait for a command to get the file again.</p>

<p>If a client doesn’t request “<b>List</b>” from the server, for a while, to check the available files, he might be looking at an old list and some files might not be available anymore. If the client requests a file that doesn’t exist, the server should reply with a message<br /> <b>&gt; File requested is not available</b><br /> Same thing should happen if a client is forwarded to another client to Get file from. If this file is not in the “Shared files” directory, the client requesting this file should receive <br /> <b>&gt;File is not available</b><br />  from the machine it was forwarded to. Both clients then close the connection between each other and send “SendMyFilesList” command to the server to update its hash table.</p>

<p> </p>

<p><b>A requirement for 5273 students:</b></p>

<p>To reduce the chance of requesting a non-existed file, the client must send “List” and “SenMyFilesList” commands periodically to the server every 3 minutes. If the server doesn’t hear from a client for 5 minutes, it should shut down the connection with that client and delete all files associated to this client from the hash table.</p>

<p><b> </b></p>

<p><b>Setting up TCP Sockets:</b></p>

<p>Your server and client must use TCP sockets, rather than UDP sockets. That means they must use send() and recv() rather than UDP's sendto() and recvfrom(). Setting up TCP sockets is more elaborate than setting up UDP sockets.</p>

<p> </p>

<p> <img src="http://moodle.cs.colorado.edu/pluginfile.php/8116/mod_assign/intro/fig2.png" width="585" height="557" alt="TCP-connection" style="display: block; margin-left: auto; margin-right: auto;" /></p>

<p align="center"><b>Figure 2 Setting up TCP Sockets</b></p>

<p>The figure above provides a pictorial description of a typical TCP client and server interaction. TCP servers are more complex to set up than TCP clients. Note your server implementation will have to loop back to accept() more often than shown, to accept() each new client that connect()'s to the server. The <a href="http://moodle.cs.colorado.edu/pluginfile.php/8116/mod_assign/intro/Sample%20PA3%20Code.rar" title="Sample PA3 Code">sample code</a>  provides an example of how to structure the server to spawn a thread after accept()'ing each new connect(). More information can be found via man pages for each of the above calls, in the lecture notes, and also at wikipedia under "socket interface".</p>

<p>Simple examples of client and server code can be found online, including at <a href="http://beej.us/guide/bgnet/output/html/singlepage/bgnet.html" title="Beej's Guide to Network Programming">Beej's Guide to Network Programming</a>.</p>

<p> </p>

<p><b>Command line</b><br /> <br /> The command-line usage of your programs should follow</p>

<p><b>./server &lt;port #&gt; </b></p>

<p>port #:is the port number the server will listen for connection on.</p>

<p><br /> <b>./client &lt;client name&gt; &lt;server ip&gt; &lt;server port #&gt; &lt;list of files from “shared files directory”&gt;</b></p>

<p>Your system must be written in C. Your client and server programs must be compiled, executed and tested on CSEL Linux computers. Files (any type of files) transferred among clients should arrive with no corruption and with the exact same size.</p>

<p> </p>

<p><b>To be turned in:</b></p>

<p>You should name your client code client.c, and create an executable called client. You should name your server code server.c, and create an executable called server.</p>

<ul>

<li>client.c code</li>

<li>“Shared files” directory contains files to be shared from the client</li>

<li>“Downloaded files” directory to download requested files in</li>

<li>server.c code</li>

<li>README</li>

</ul>

<p> </p>

<p><b>Grading:</b></p>

<p>You should submit all of the above to the Moodle by the deadline. The code should be well-documented enough so that we can verify that the server and client work correctly. The README should briefly explain the structure of your system and explain which of the following aspects of the assignment work or do not work:</p>

<ul>

<li>clients are able to connect to the server via TCP and execute the commands above</li>

<li>clients are able to send and receive TCP messages and send files to each other</li>

<li>server correctly routes all file requests  to the correct clients</li>

<li>server gracefully close connection with clients that do not update for 5 minutes (for 5273 students only0</li>

</ul>
